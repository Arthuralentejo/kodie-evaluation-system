## Pergunta 3

**Categoria:** iniciante
**Tags:** [hardware, memoria, fundamentos]

**Pergunta:**
O que diferencia a memória RAM do armazenamento em disco (HD/SSD)?

**Alternativas:**
a) A RAM é mais lenta, porém armazena dados permanentemente
b) A RAM é volátil e rápida, perdendo seus dados ao desligar o computador
c) O disco (HD/SSD) é mais rápido que a RAM para leitura de dados

## Pergunta 17

**Categoria:** junior
**Tags:** [linux, permissoes, cli]

**Pergunta:**
No Linux, o que representa a permissão `chmod 755 arquivo.sh`?

**Alternativas:**
a) O dono pode ler, escrever e executar; grupo e outros podem apenas ler e executar
b) Todos os usuários têm permissão total de leitura, escrita e execução
c) O dono pode apenas ler; grupo pode ler e executar; outros não têm acesso
d) O arquivo é protegido contra leitura por qualquer usuário

**Resposta correta:** a

**Explicação:**
Em notação octal, `7` corresponde a `rwx` (leitura + escrita + execução) e `5` corresponde a `r-x` (leitura + execução). No `chmod 755`: o dono (7) tem permissão total; o grupo (5) pode ler e executar; outros (5) também podem ler e executar. Essa é uma configuração comum para scripts executáveis.

## Pergunta 29

**Categoria:** pleno
**Tags:** [banco-de-dados, sql, performance]

**Pergunta:**
Qual a finalidade principal de um índice em um banco de dados relacional?

**Alternativas:**
a) Garantir a integridade referencial entre tabelas por meio de chaves estrangeiras
b) Aumentar a velocidade de busca em colunas frequentemente consultadas, ao custo de maior espaço em disco e tempo de escrita
c) Definir o esquema e os tipos de dados de cada coluna de uma tabela
d) Controlar o nível de isolamento das transações simultâneas

**Resposta correta:** b

**Explicação:**
Índices são estruturas de dados auxiliares (geralmente B-trees) que aceleram consultas em colunas específicas, evitando full table scans. O trade-off é que ocupam espaço adicional em disco e tornam operações de escrita (INSERT, UPDATE, DELETE) mais lentas, pois o índice também precisa ser atualizado. Devem ser usados criteriosamente em colunas de alta seletividade e frequente consulta.

## Pergunta 32

**Categoria:** pleno
**Tags:** [solid, orientacao-a-objetos, design]

**Pergunta:**
Qual é o objetivo do princípio "Dependency Inversion" do SOLID?

**Alternativas:**
a) Garantir que dependências entre classes sejam sempre injetadas via construtor, nunca via setter
b) Fazer com que módulos de alto nível não dependam de módulos de baixo nível, ambos devendo depender de abstrações (interfaces)
c) Proibir que classes utilizem bibliotecas externas diretamente, exigindo wrappers intermediários
d) Inverter a ordem de inicialização de objetos para otimizar o uso de memória

**Resposta correta:** b

**Explicação:**
O **Dependency Inversion Principle** (DIP) estabelece que: (1) módulos de alto nível não devem depender de módulos de baixo nível — ambos devem depender de abstrações; (2) abstrações não devem depender de detalhes — detalhes devem depender de abstrações. Isso permite trocar implementações sem alterar o código que as utiliza, facilitando testes e manutenção.

**Categoria:** sênior
**Tags:** [python, asyncio, concorrencia, race-condition]

**Pergunta:**
Analise o código Python abaixo e identifique o problema de concorrência:

```python
contador = 0

async def incrementar():
    global contador
    valor_atual = contador
    await asyncio.sleep(0)  # cede controle ao event loop
    contador = valor_atual + 1

async def main():
    await asyncio.gather(*[incrementar() for _ in range(100)])
    print(contador)  # Qual valor esperado?
```

**Alternativas:**
a) O código é thread-safe pois usa `asyncio`, que é single-threaded; o resultado será sempre `100`
b) Há uma race condition: `await asyncio.sleep(0)` cede o controle, permitindo que outras corrotinas leiam `contador` antes da escrita, resultando em valor menor que `100`
c) O código lança `RuntimeError` pois variáveis globais não podem ser modificadas em corrotinas assíncronas
d) O resultado será `100` na maioria das execuções, mas pode variar por causa do GIL do Python

**Resposta correta:** b

**Explicação:**
Mesmo sendo single-threaded, `asyncio` tem pontos de troca de contexto nos `await`. Quando `incrementar` lê `contador=0` e cede o controle com `await asyncio.sleep(0)`, outras 99 corrotinas também leem `contador=0` antes de qualquer escrita. Todas escrevem `0+1=1`. O resultado final será próximo de `1`, não `100`. A solução é usar `asyncio.Lock` para proteger a seção crítica.

**Pergunta:**
Você precisa desenhar um sistema de rate limiting para uma API pública que permite 100 requisições por minuto por usuário, distribuído em 10 instâncias de API. Analise as duas abordagens:

**Abordagem A:** Cada instância mantém um contador local em memória.
**Abordagem B:** Todas as instâncias incrementam um contador centralizado no Redis com TTL de 60 segundos.

Qual é o trade-off correto entre elas?

**Alternativas:**
a) A é mais precisa; B tem latência adicional de rede que torna o rate limiting impraticável em produção
b) A permite até 1000 req/min por usuário (100 × 10 instâncias) sem que nenhuma instância perceba a violação; B é precisa mas adiciona uma dependência de infraestrutura e latência de rede
c) Ambas são equivalentes se o load balancer usar sticky sessions para rotear sempre o mesmo usuário para a mesma instância
d) A é a abordagem recomendada pois evita single point of failure; B cria gargalo centralizado que derruba todas as instâncias se o Redis cair

**Resposta correta:** b

**Explicação:**
Com contadores locais, cada instância permite 100 req/min independentemente — um usuário pode fazer 100 req em cada uma das 10 instâncias = 1000 req/min sem ser bloqueado. Redis centralizado garante precisão, mas adiciona ~1ms de latência por requisição e cria dependência. Na prática, usa-se Redis com algoritmos como **sliding window** ou **token bucket**, aceitando o trade-off de latência em troca de precisão. Sticky sessions ajudam, mas falham quando instâncias escalam ou reiniciam.

## Pergunta 76

**Categoria:** sênior
**Tags:** [system-design, banco-de-dados, performance, particionamento]

**Pergunta:**
Um sistema legado usa uma única tabela `eventos` com 2 bilhões de registros. Queries por intervalo de data levam 40 segundos. O DBA propõe particionar a tabela por mês. Um engenheiro sênior sugere também avaliar uma estratégia de arquivamento. Por que a sugestão do engenheiro é relevante?

**Alternativas:**
a) Particionamento por mês cria automaticamente 24 tabelas separadas, o que excede o limite de tabelas do PostgreSQL
b) Particionamento melhora queries que filtram por data, mas a tabela continua com 2B registros; arquivar partições antigas em cold storage reduz o volume ativo, melhorando cache hit rate e manutenção
c) O arquivamento é necessário porque particionamento só funciona em tabelas com menos de 1 bilhão de registros
d) Particionamento e arquivamento são estratégias mutuamente exclusivas — implementar ambas causa inconsistência nos dados

**Resposta correta:** b

**Explicação:**
Particionamento melhora o pruning de partições (queries só varrem partições relevantes), mas o volume total de dados permanece. Com 2B registros, mesmo particionado, partições antigas ocupam espaço, degradam backups e aumentam o tempo de VACUUM/ANALYZE. Arquivar partições antigas (ex: dados com mais de 1 ano) em object storage (S3 + Parquet) ou tabelas de arquivo reduz o conjunto ativo, melhora o cache do buffer pool e simplifica manutenção — complementando, não substituindo, o particionamento.
